# BlogAPI

RESTful blog backend built with **ASP.NET Core 10**, following **Clean architecture**. Supports posts, tags, comments and user profiles with JWT Bearer authentication. Read operation are public. Write operations require authentication.

Was build as a portfolio project to demonstrate usage of Clean Architecture, the Result pattern, Identity auth, EF Core with PostgreSQL, unit tests with NSubstitute, and full integration tests using Testcontainers.

## Features
- **JWT Authentication** - Register, login and manage credentials (username, email, password)
- **Posts** - Create, read, update, delete blog posts with autogenerated or user specified URL slugs
- **Tags** - Many-to-many tagging on posts, searchable tag index
- **Comments** - Comment threads per post
- **Filtering & sorting** - List endpoints support query parameters for filtering and sorting
- **Pagination** - All list endpoints return paginated results with default or user specified page sizes
- **Validation** - FluentValidation checks on incoming DTOs
- **RFC 7807 Problem Details** - Consistent error responses accross all endpoints
- **Structured logging** - Serilog daily file output
- **Swagger UI** - Interactive API documentation in development with auth implemented
- **Email confirmation** - Registration generates a confirmation token; Resend endpoint is provided for expired or lost tokens.
- **Password reset & email change** - Forgot-password and email-change flows generate single-use tokens delivered via the email outbox.
- **Transactional email delivery** - Outbound emails are persisted as `OutboxMessage` rows and dispatched by a `BackgroundService` (`OutboxEmailProcessor`) that polls in batches with `FOR UPDATE SKIP LOCKED`, retries failures up to a configured maximum, and records the last error per row for diagnostics.

## Architecture

The project follows **Clean Architecture** with four layers, enforcing dependency rules (inner layers have no knowledge of outer layers):

```
BlogAPI.Domain          ← Entities, domain interfaces, Result/Error types. No dependencies.
BlogAPI.Application     ← Services, DTOs, validators, mappers, query filters/sorters. Depends on Domain only.
BlogAPI.Infrastructure  ← EF Core DbContext, repositories, JWT service, ASP.NET Identity. Depends on Application + Domain.
BlogAPI (Web)           ← Controllers, middleware, DI wiring. Depends on all layers.
```


---

### Key patterns

**Result pattern** — All service methods return `Result` or `Result<T>` instead of throwing exceptions. Controllers call `.ToProblemDetails()` to convert failures into RFC 7807 responses. Error types: `NotFound`, `Validation`, `Conflict`, `Forbidden`, `Unauthorized`, `Internal`.

**Repository pattern** — Infrastructure repositories implement domain interfaces, keeping EF Core out of the application layer.

**Query pipeline** — List endpoints compose `IQueryFilter<T>` and `IQuerySorting<T>` implementations over `IQueryable<T>`, then materialise a `PagedList<T>` via `IPagedListFactory`.

**Dual identity** — `ApplicationUser` (ASP.NET Core Identity) handles auth mechanics. `UserProfile` is the domain entity that owns posts and comments. They are linked via `UserProfile.ApplicationUserId`. `IUserContext` exposes the current user's IDs from JWT claims.

## Getting Started

### Prerequisites
- [Docker](https://www.docker.com/get-started) (for running with Docker Compose)
or
- [.NET 10 SDK](https://dotnet.microsoft.com/en-us/download/dotnet) (for local dev)
- 
### Run with Docker Compose
Easly run the full application — no local .NET SDK required.

**1. Clone the repository**

```bash
git clone https://github.com/RedRabel99/BlogAPI.git
cd BlogAPI
```

**2. Set required secrets**

The app reads JWT configuration from environment variables. Create a `.env` file in the repository root:

```env
JWT__Secret=your-super-secret-key-at-least-32-characters-long
JWT__Issuer=myApp
JWT__Audience=myApp
```

**3. Start the stack**

```bash
docker compose up --build
```

This spins up:
- `db` — PostgreSQL on port `5432` (credentials: user =`postgres` password = `postgres`, database: `blogapi-db`)
- `api` — ASP.NET Core API (once the container is added to the compose file)

**4. Access the API**

- API: `http://localhost:8080`
- Swagger UI: `http://localhost:8080/swagger` (development mode)

> **Note:** EF Core migrations run automatically on startup.

---

### Run Locally (without Docker)
**1. Start PostgreSQL**
```bash
docker compose up postgres -d
```

**2. Configure user secrets**

```bash
dotnet user-secrets set "JWT:Secret" "your-super-secret-key-at-least-32-characters" --project BlogAPI
dotnet user-secrets set "JWT:Issuer" "myApp" --project BlogAPI
dotnet user-secrets set "JWT:Audience" "myApp" --project BlogAPI
```

**3. Apply migrations**

```bash
dotnet ef database update --project BlogAPI.Infrastructure --startup-project BlogAPI
```

**4. Run the API**

```bash
dotnet run --project BlogAPI/BlogAPI.Web.csproj
```

## Example Usage

All examples use `curl`. 

### Register and login

```bash
# Register
curl -X POST http://localhost:8080/auth/register \
  -H "Content-Type: application/json" \
  -d '{
    "username": "johndoe",
    "displayName": "John Doe",
    "email": "john@example.com",
    "password": "Password1!"
  }'

# Login — returns a JWT token
curl -X POST http://localhost:8080/auth/login \
  -H "Content-Type: application/json" \
  -d '{
    "email": "john@example.com",
    "password": "Password1!"
  }'
# Response: { "token": "eyJ..." }
```

Store the token for authenticated requests:

```bash
TOKEN="eyJ..."
```

### Create and retrieve a post

```bash
# Create a post
curl -X POST http://localhost:8080/post \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "title": "My First Post",
    "content": "Hello, world!",
    "excerpt": "A short intro.",
    "tags": ["dotnet", "clean-architecture"]
  }'

# Get all posts (paginated)
curl "http://localhost:8080/post?page=1&pageSize=10"

# Get a specific post
curl "http://localhost:8080/post/3fa85f64-5717-4562-b3fc-2c963f66afa6"
```

### Comment on a post

```bash
# Add a comment
curl -X POST "http://localhost:8080/posts/3fa85f64-5717-4562-b3fc-2c963f66afa6/comments" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{ "content": "Great post!" }'

# List comments on a post
curl "http://localhost:8080/posts/3fa85f64-5717-4562-b3fc-2c963f66afa6/comments?page=1&pageSize=20"
```

### Error response format

All errors follow RFC 7807 Problem Details:

```json
{
  "type": "https://tools.ietf.org/html/rfc7807",
  "title": "Validation Error",
  "status": 400,
  "errors": [
    { "field": "content", "message": "Content must not be empty." }
  ]
}
```

## Design Decisions

**Result pattern over exceptions** — Service methods return `Result<T>` rather than throwing. This makes failure paths explicit, keeps controller code uniform, and avoids the performance expensive exceptions for expected failures (e.g. 404, 409). Errors are mapped to adequate proble details.

**Dual user model** — ASP.NET Core Identity (`ApplicationUser`) manages credentials and auth mechanics. The domain `UserProfile` entity owns posts and comments and carries no Identity concerns. This separation keeps the domain clean and makes the Identity layer swappable.

**Flat comments** — Comments are top-level replies on a post with no nesting. This keeps the data model and API simple; threaded replies are a planned future feature. Possible implementation of nesting in the future.

**Hard deletes** — Neither posts nor comments use a soft-delete flag. Cascade delete on Post → Comments is configured at the database level. This avoids filter complexity at the cost of auditability (a trade-off for this small project).

**Ownership-based authorisation** — Edit/delete operations verify that the requesting `UserProfileId` matches the resource owner. Admin moderation will be implemented when a role system is added.

**Composable query pipeline** — Filtering and sorting are encapsulated in `IQueryFilter<T>` / `IQuerySorting<T>` implementations applied over `IQueryable<T>`. Adding a new filter or sort criterion requires only a new implementation — no changes to the repository or controller. Query is send to database after filters and sorting is applied.

**Pagination with default values** - Pagination is applied before query is sent to the database. This improves performance and avoids loading unnecessary data into memeory. `PagedListFactory` enforces page size constraints and falls back to defaults defined via IOptions<PaginationOptions>, which are configured in appsettings.json.

**Integration tests with real database** — Integration tests run against a real PostgreSQL container (Testcontainers) rather than an in-memory database. This catches migration issues and query behaviour that in-memory providers would miss.

**Outbox pattern for email delivery** — Email payloads are serialised into an `OutboxMessages` table and processed asynchronously by `OutboxEmailProcessor`, a hosted `BackgroundService`. The processor claims rows under `FOR UPDATE SKIP LOCKED`, deserialises into the `EmailMessage` contract, sends through `IEmailSender`, and either stamps `ProcessedOn` on success or increments `RetryCount` and stores the error on failure. This decouples user registration from email delivery and persists an email in case of SMTP or service errors.
